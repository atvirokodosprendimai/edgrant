# Formal Verification Model

## Target

A machine-checked proof that the EdGrant grant exchange is secure under a network-controlling adversary. The model covers the three security-critical phases (request, grant, access) and proves seven properties about the protocol's behaviour.

## Behaviour

### What the Model Proves

- **Request authenticity** — if a grant was issued, the requestor actually made that exact request (same resource, scopes, TTL). An attacker cannot trick the resource owner into granting something the requestor didn't ask for
- **No scope escalation** — the scopes in the received grant match the scopes the requestor originally requested. No party can widen scopes between request and grant
- **Bot binding** — a grant is cryptographically bound to exactly one requestor key. Two different keys cannot share a grant, even if an attacker controls the network
- **No request forgery** — an attacker cannot forge a grant request on behalf of a requestor without possessing the requestor's private key
- **Grant secrecy** — the attacker cannot learn the capability token. The grant delivery channel protects token confidentiality
- **Grant authorization** — only a resource owner (holder of a resource owner key pair) can issue a valid grant. No other party can produce one
- **Executability** — the protocol can actually complete successfully. This is a non-vacuity check: without it, the other six properties could be trivially true because no valid trace exists

### What the Model Excludes (by design)

- **Discovery (Phase 0)** — informational exchange only, no secrets or signatures involved. No security properties to verify
- **Policy evaluation** — the resource owner's decision logic is abstracted as a precondition. If a grant rule fires, policy was satisfied. What policy contains (bot owner approval, risk assessment, scope limits) is outside the model's scope
- **Scope semantics** — scopes are opaque fresh values. The model proves scopes are preserved, not that they mean anything specific
- **Token expiry** — TTL enforcement is a runtime concern. The model proves the TTL value is preserved from request to grant, not that clocks are checked
- **Revocation** — a registry concern that operates after grant issuance, outside the grant exchange

### Adversary Model

- **Dolev-Yao** — the attacker controls the entire network: can intercept, modify, replay, and inject messages
- The attacker knows all public keys (published by key generation rules)
- The attacker does NOT know private keys or TLS session keys (fresh, never published)
- This is the standard symbolic adversary for protocol verification — if properties hold under Dolev-Yao, they hold against any network-level attacker

## Design

### TLS Idealisation

- TLS is modelled as symmetric encryption with a fresh key shared via internal state (never on the public channel)
- This abstracts away the TLS handshake — the model assumes TLS itself is secure
- The fresh key (`~tls_key`) is generated by the requestor and consumed by the resource owner via a state fact, preventing the attacker from accessing it
- {>> if TLS-level attacks are in scope, a separate TLS model would be needed — but that's a TLS concern, not an EdGrant concern}

### Namespace Separation

- Request signatures use namespace constant `'grant_request'`
- Grant signatures use namespace constant `'resource_owner_grant'`
- These string tags are part of the signed message, preventing an attacker from replaying a request signature as a grant signature (or vice versa)
- Also prevents cross-protocol replay with EdProof's `'edproof'` namespace

### Collision Resistance Assumption

- `fingerprint/1` is modelled as an injective function in the free term algebra
- This assumes SHA-256 is collision-resistant — two different public keys cannot produce the same fingerprint
- The `bot_binding` lemma depends on this assumption
- {>> if SHA-256 collisions become practical, this assumption breaks and bot binding is no longer guaranteed — but this would affect the entire EdProof ecosystem, not just EdGrant}

### Lemma Overlap

- Lemmas 1 (`request_authenticity`) and 4 (`no_request_forgery`) prove the same underlying property from different perspectives
- This is intentional: authenticity says "grants trace back to real requests", forgery says "attackers can't fake requests"
- Same proof basis (signature verification), different framing for different audiences (protocol designers vs security reviewers)

### Verification Performance

- All seven lemmas verified in 0.66 seconds
- Fast verification suggests a clean model with no state explosion — the protocol is structurally simple
- Requires: Tamarin prover ≥ 1.10, Maude ≥ 3.5

## Interactions

- **Implements** the security claims in [[eidos/spec - edgrant protocol.md]] §Formal Verification
- **Models** the grant exchange from [[RFC-EDGRANT.md]] §5-7 (request, grant, access)
- **Assumes** EdProof identity is established (not re-verified — see separate edproof.spthy)
- **Does not model** discovery, policy, revocation, or scope semantics — these are outside the security perimeter of the grant exchange
